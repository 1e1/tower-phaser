Pathfinding Lab — Tower Duel · le graphe de navigation des soldats (dérivé de battlefield-lab)

Tu connais déjà ce graphe : c'est l'oscilloscope que le 3ᵉ joueur (l'Intendant) a sous les yeux sur sa manette. Chaque segment du terrain y est colorié selon sa franchissabilité par pente. Le « Debug chemins » des développeurs est devenu une aide de jeu. Ici, on le déplie : pose des planches, vois les nœuds du graphe, le chemin A*, et creuse un cratère pour regarder le navVer s'incrémenter et la replanification se déclencher.

Lecture. Le graphe est implicite : un nœud = un couple (colonne, surface) renvoyé par surfacesAt(x) (le sol, plus chaque planche au-dessus). Une arête relie deux colonnes voisines si la marche est franchissable (montée ≤ SU) ; les descentes passent toujours mais coûtent plus cher (pénalité de chute). findPath applique Dijkstra/A* du soldat vers la tour adverse ; s'il n'y a aucun chemin, le soldat déserte.
Replanification. Chaque édition (pose d'ouvrage, cratère) incrémente navVer. Les soldats comparent leur navVer mémorisé à l'actuel et recalculent ; un replan périodique (~0,7 s) absorbe en plus les éditions continues (creuser/aplanir). Ici, chaque cratère bump navVer → tu vois le chemin se redessiner.
